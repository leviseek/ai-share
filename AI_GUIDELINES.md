# AI 协作规范

本文档是用户级通用 AI 协作规范，适用于不同电脑、不同项目中的 OpenCode 使用场景。

## 基本原则

- 优先理解现有项目结构、代码风格和约定，再进行修改。
- 默认做最小正确改动，避免不必要的重构、抽象和兼容层。
- 不擅自覆盖、回滚或删除用户已有改动。
- 不把密钥、令牌、私有凭据写入代码、文档或提交记录。
- 遇到关键不确定性时，先提出简短问题；无需确认的小问题应自行判断并继续推进。

## 工作方式

- 修改代码前，先读取相关文件并建立上下文。
- 多步骤任务应拆解后持续推进到可验证状态。
- 修改完成后尽量运行与改动相关的检查、测试或构建命令。
- 如果无法验证，应明确说明原因和剩余风险。
- 发现与当前任务无关的脏工作区改动时，只记录或避开，不主动处理。

## 代码修改

- 保持改动范围聚焦，避免顺手修改无关文件。
- 遵循项目已有命名、目录结构、格式化和类型约束。
- 新增逻辑应优先简单直接，只有复用价值明确时才抽取工具函数或模块。
- 注释只用于解释不明显的意图、限制或复杂逻辑，不重复描述代码表面行为。
- 默认使用 ASCII；只有文件已有中文或业务需要中文时才使用中文内容。

## 命令与文件

- 执行破坏性命令前必须获得明确授权，例如强制删除、强制推送、硬重置等。
- 不主动提交 Git，除非用户明确要求。
- 不在仓库内生成或提交本地密钥文件，例如 `.env`、凭据 JSON、私钥等。
- 修改配置时优先使用环境变量引用敏感值。

## Git 提交规范

- 只有在用户明确要求提交时才执行 `git commit`。
- 提交前应检查 `git status` 和 `git diff`，确认只提交与任务相关的文件。
- 不提交密钥、令牌、本地环境文件、临时产物或无关改动。
- Git 命令 options 使用英文，例如 `git commit -m`、`git status --short`、`git diff --staged`。
- 提交信息遵循 `GIT_COMMIT_GUIDELINES.md`。
- 提交信息格式为 `option: 中文描述`，其中 `option` 使用英文小写，`:` 后的描述内容使用中文。
- 示例：`config: 补充 OpenCode 共享模型提供商`。
- 不使用 `--no-verify`、`--amend`、强制推送等跳过或改写历史的操作，除非用户明确要求。

## AI 配置约定

- OpenCode 主配置使用 `opencode.jsonc`，不使用 YAML 作为直接加载配置。
- 共享模型提供商、模型列表和默认模型应维护在本仓库的 `opencode.jsonc`。
- 每个具体项目可以放置自己的 `opencode.jsonc` 做少量覆盖。
- API Key 只通过环境变量引用，例如 `{env:DEEPSEEK_API_KEY}`。
- 变更模型提供商、`baseURL`、API Key 环境变量名或模型列表时，优先使用 `bun run check:ai`。
- 将共享配置安装到当前电脑时，优先使用 `bun run share`。

## 输出风格

- 用中文直接说明结果、改动点和验证情况。
- 简单任务简短回复；复杂任务按“结果、改动、验证、后续建议”组织。
- 不输出冗长的思维过程，只给出必要依据和可执行信息。
- 如果任务是代码审查，应优先列出问题、风险和对应文件位置。
