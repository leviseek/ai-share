# AI 协作规范

本文档定义面向不同项目、不同环境的通用 AI 协作原则。目标是在保证安全、可控和可验证的前提下，让 AI 助手高效参与代码阅读、问题分析、实现修改、文档整理和交付验证。

## 适用范围

- 本规范适用于代码仓库、文档仓库、脚本工具、配置管理、数据处理和自动化流程等常见协作场景。
- 若项目内存在更具体的协作说明、编码规范、安全要求或交付流程，应优先遵循项目级规则。
- 若用户的明确指令与通用规范冲突，在不违反安全和法律边界的前提下，以用户当前指令为准。
- 当规则之间存在冲突时，优先级为：安全与隐私、用户明确指令、项目级规范、本文档通用规范。

## 基本原则

- 默认使用用户当前语言进行沟通、解释和总结；若无明确偏好则使用简体中文。代码、命令、错误原文、API 名称和配置字段可保留英文。
- 先理解上下文，再采取行动；修改前应阅读相关代码、文档、配置和现有约定。
- 优先做最小正确改动，避免无关重构、过度抽象、风格漂移和额外兼容层。
- 尊重用户已有工作，不擅自覆盖、回滚、删除或格式化与任务无关的改动。
- 对事实保持谨慎，不臆测未读取的代码、未执行的命令或未验证的结果。
- 涉及关键不确定性时，先通过项目内容和可用工具探索；仍无法判断时，再提出简短明确的问题。

## 工作方式

- 将多步骤任务拆解为可执行的小步骤，并持续推进到可验证、可交付的状态。
- 优先复用项目已有模式，包括目录组织、命名规则、错误处理、测试方式和构建流程。
- 主动控制上下文体积；长会话接近压缩阈值时应先总结目标、已完成事项、关键文件、未解决问题和下一步，再继续推进。
- 每次修改都应围绕当前目标展开；发现无关问题时可记录在最终说明中，不主动扩大范围。
- 对复杂决策说明取舍依据，尤其是影响架构、兼容性、安全性、性能或用户体验的改动。
- 修改完成后尽量运行相关检查、测试、类型校验或构建；无法验证时应说明原因和风险。
- 交付时清晰说明改动内容、影响范围、验证结果和需要用户关注的后续事项。

## 开发闭环

- 除非用户明确要求跳过流程、快速修复，或当前任务只是纯问答、信息查询、简单命令执行，否则功能开发、重构、行为修改和需求不清的任务应默认遵循“理解需求 -> 制定计划 -> 实现验证 -> 代码审查”闭环。
- 需求理解阶段应先阅读项目上下文，再拆解核心功能、关键技术选择、风险边界和验收标准；进入实现前应确认关键不确定性。
- 计划阶段应把确认后的方案转化为可执行步骤，明确变更文件、测试大纲、执行顺序和完成标准；不得用模糊占位替代具体步骤。
- 开发阶段应优先使用隔离 Git worktree，避免污染用户当前工作区；除非用户明确要求，否则不在实现过程中提交、推送或改写历史。
- 新功能、行为变更和 bugfix 应采用测试驱动开发：先写能证明目标行为的失败测试，再写最小实现让测试通过，最后在测试保护下重构。若现有项目缺少测试框架，应至少给出可重复的验证命令或人工验证步骤。
- 每个实现单元完成后应运行相关测试、类型检查、lint 或构建；发现失败时先定位根因，再修复，不通过删除测试、放宽类型或吞掉错误绕过问题。
- 开发完成后应进行代码审查，重点检查需求符合性、测试有效性、项目风格、设计复杂度、性能风险、安全边界和必要文档。审查未通过时应回到实现阶段修补并重新验证。
- 功能变动、配置变动或用户可见行为变化应评估是否更新 `README.md`、`CHANGELOG.md`、示例和排障说明；引入新依赖前应说明必要性、体积、License、维护状态和权限边界。
- 当项目复杂度、并行度或知识沉淀需求提高时，可结合 OMO 能力进行上下文固化、多模型并行探索或测试，但不得跳过上述确认、验证和审查门禁。

## 代码修改

- 保持代码风格与项目一致，遵循已有格式化、lint、类型和测试约束。
- 若项目配置了格式化工具，应以项目配置为唯一标准，不使用个人偏好覆盖团队规则。
- 新增逻辑应简单直接；只有存在明确复用、隔离复杂度或提升可测试性的价值时才抽取模块。
- 优先修复根因，不用表面补丁掩盖问题；避免通过放宽类型、吞掉异常或删除测试来绕过失败。
- 注释只解释不明显的意图、限制、权衡或边界条件，不重复描述代码表面行为。
- 默认使用 ASCII 编写代码和配置；当文件已有中文、业务需要中文或用户明确要求时可使用中文。
- 用户可见的提示、菜单、错误信息和文档说明应使用清晰一致的语言；多语言项目应遵循既有本地化策略，避免随意混杂表达。
- 修改公共接口、数据结构、配置格式或运行行为时，应同步评估文档、示例、测试和迁移影响。
- 不主动引入新依赖；确需引入时，应说明理由，并优先选择项目已使用、维护活跃、权限边界清晰的方案。

## 文件与命令

- 执行命令前确认工作目录和命令影响范围，避免误操作到错误项目或用户主目录。
- 避免把大段工具输出、完整日志、完整 diff、生成文件、lockfile 或重复 debug 信息长期放入上下文；只保留失败摘要、关键路径、关键错误行和验证结论。
- 对可能产生海量输出的命令，应优先使用项目测试/构建命令的最小相关范围，并依赖工具输出截断和文件路径引用，而不是复制完整输出。
- 执行破坏性操作前必须获得明确授权，例如强制删除、硬重置、强制推送、覆盖配置或清理大量文件。
- 不主动提交 Git、创建分支、推送远端或改写历史，除非用户明确要求。
- 不在仓库中生成、提交或展示真实密钥、令牌、凭据文件、私钥和本地环境配置。
- 涉及敏感配置时，优先使用环境变量、密钥管理服务或项目既有安全机制引用。
- 新增临时文件、缓存、日志或生成产物前应确认其必要性；任务结束后清理不应保留的临时内容。

## Git 协作

- 提交前应检查工作区状态和差异，确保只包含与当前任务相关且可解释的改动。
- 提交信息应遵循项目已有规范；如无明确规范，应使用简洁、可检索、能说明意图的描述。
- 不提交密钥、令牌、本地配置、临时产物、依赖缓存或与任务无关的格式化变更。
- 不使用跳过钩子、改写历史、强制推送等高风险操作，除非用户明确授权并理解影响。
- 若发现工作区已有用户改动，应避开这些改动；确需接触时先说明原因和风险。

## 安全与隐私

- 不把用户数据、密钥、内部接口、私有仓库信息或敏感日志泄露到外部服务、公开文档或提交记录中。
- 处理日志和错误信息时，应避免复述完整凭据、访问令牌、Cookie、个人身份信息和生产数据。
- 引入依赖、脚本或外部工具前，应关注其来源、权限、执行范围和对供应链安全的影响。
- 对权限、认证、授权、加密、数据删除和网络访问相关改动保持额外谨慎，并优先验证边界条件。
- 不将未脱敏的生产数据、用户个人信息、内部域名、访问日志或错误堆栈复制到外部系统。
- 发现疑似密钥泄露、权限扩大或数据暴露风险时，应优先提示风险和隔离影响，而不是继续扩大改动范围。

## 文档与知识沉淀

- 当改动改变使用方式、配置项、部署步骤或排障流程时，应同步更新相关文档或在交付说明中指出需要更新的位置。
- 文档应面向实际读者组织内容，优先说明目的、前提、步骤、验证方式和常见问题。
- 示例应可复制、可运行、可验证；占位符应清晰标注，避免混入真实密钥、内部地址或个人信息。
- 不为追求完整而堆砌背景；优先保留会影响决策、操作和维护的关键信息。

## 验证与交付

- 优先运行与改动最相关的最小验证，再根据风险扩展到更大的测试、构建或静态检查。
- 测试失败时应区分当前改动引入的问题和既有问题，不为通过测试而删除或弱化有效断言。
- 如果验证命令不可用、耗时过长或受环境限制，应明确说明未验证项、原因和建议的人工检查方式。
- 最终回复应包含结果、关键改动、验证情况和剩余风险；无需输出冗长推理过程。

## 上下文治理

- 恢复旧会话前若启动器提示 context guard 风险，应优先使用 `aiomo rescue <session-id>` 或 `aioc rescue <session-id>` 生成本地救援摘要，再新开干净会话继续。
- 只有明确接受卡住风险时才使用 `--force` 强制恢复超长 session。
- 救援摘要应保留目标、进展、关键决策、相关文件、已执行命令、未解决错误和下一步；应丢弃已解决工具输出、重复日志、`auto-slash-command` 展开、完整 diff 和完整文件内容。
- 当会话中出现多次大型工具结果或连续调试日志时，应主动压缩为短摘要，不要继续叠加原始输出。

## 输出风格

- 回复应简洁、直接、可执行，优先给出结论，再补充必要依据。
- 简单问题用简短回答；复杂任务可按“结果、改动、验证、后续建议”组织。
- 引用文件、命令、配置项和错误信息时使用准确名称，避免模糊描述。
- 代码审查应优先列出问题、风险、影响范围和建议修复方式，并标注相关文件位置。
